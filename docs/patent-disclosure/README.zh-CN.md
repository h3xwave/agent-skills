# 专利交底书 Skill

`patent-disclosure` 是一个用于专利交底书撰写的 Codex skill，可将发明想法、技术笔记、研发方向或已有技术材料整理为结构化专利交底材料。该 skill 默认面向中国专利实践，通常输出中文 Markdown 交底书正文，同时支持可申请方向挖掘、现有技术冲突预判、结构化访谈、已有交底书修改补强，以及 Word 文档导出。

英文文档见 [README.md](README.md)。

该 skill 不替代专利代理师、律师或正式法律意见。它的作用是帮助发明人形成可提交给企业知识产权部门或专利代理机构审阅的技术材料。

## 适用场景

当用户提出以下请求时，使用该 skill：

- 起草专利交底书。
- 将发明想法整理为正式交底材料。
- 从宽泛技术方向中挖掘可申请切入点。
- 评估授权前景、撞车风险或现有技术冲突。
- 修改、补强或润色已有交底书。
- 生成或导出 Word 格式 `.docx` 交底书。

典型中文请求包括：

```text
帮我写专利交底书
我有个发明想申请专利
帮我把技术方案整理成交底书
生成技术交底
评估授权前景
把这份交底书导出成 Word
```

## 能力范围

该 skill 支持五类主要任务：

1. **方向挖掘**
   - 适用于用户只有宽泛技术领域或模糊想法的情况。
   - 输出三个具有授权潜力的切入点，并说明可保护对象、区别特征、主要风险和需要补充的信息。

2. **结构化访谈**
   - 适用于发明方案尚未完整描述的情况。
   - 按顺序收集技术问题、现有技术、现有缺点、技术方案、附图和创新点。

3. **授权前景预判**
   - 用于正式起草前的现有技术风险评估。
   - 检索相关专利和论文，比较高风险对比文件，并输出《专利申请预判报告》。

4. **交底书起草与修改**
   - 按标准五章结构生成中文交底书。
   - 也可以对已有交底书进行补强、润色和结构化改写。

5. **Word 导出**
   - 使用内置 Python 脚本将完整 Markdown 交底书转换为 `.docx` 文件。

## 必须遵循的流程

该 skill 采用分阶段门槛流程。除非用户明确只要求对已有文本做局部润色，并且已说明风险，否则不应跳过前置阶段。

### 第一道门槛：公开风险确认

正式起草前必须询问发明是否已经通过以下方式公开：

- 论文或预印本。
- 会议或公开演讲。
- 产品发布或宣传页面。
- 招投标材料或客户方案。
- 开源仓库。
- 文章、博客、公众号或社交媒体。

如果用户回答“不清楚”“不知道”等，应按公开状态不明处理，并仅使用抽象后的通用技术词进行公开风险检索。除非用户明确确认可以使用，否则不要把内部产品名、客户名、精确参数、源代码或未公开完整方案作为搜索词。

### 第二道门槛：信息收集

如果用户没有提供完整技术方案，应按访谈流程依次收集：

1. 技术问题与背景。
2. 现有技术缺点与发明目的。
3. 技术方案、模块、步骤、输入、输出和附图。
4. 创新点和技术效果。

每轮提问不超过三个问题。

如果用户已经提供完整文档、技术说明或交底书草稿，则走“已有材料路径”：

- 将材料映射到交底书五个章节。
- 标注已覆盖、部分覆盖、缺失或需确认的内容。
- 一次最多追问三个关键缺口问题。

### 第三道门槛：授权前景预判

正式起草前应输出《专利申请预判报告》，内容包括：

- 检索截止日期。
- 检索目标。
- 检索策略、数据库、关键词、分类号和时间窗口。
- 核心概念界定。
- 现有技术冲突检索结果。
- 学术文献参考。
- 乐观、基准、悲观三种授权情景。
- 影响授权前景的有利和不利因素。
- 提高授权前景的策略建议。
- 是否建议进入起草的结论。

报告必须说明其不是法律意见，也不承诺授权结果。

### 第四道门槛：用户确认

只有在用户确认预判报告，或明确回复“继续”等进入下一步的指令后，才进入正式交底书起草。

### 第五道门槛：正式起草

交底书必须使用固定五章结构：

1. 本发明要解决的技术问题是什么。
2. 背景技术。
3. 现有技术缺点推导及本发明目的说明。
4. 技术方案深度构建。
5. 本发明的关键点和欲保护点是什么。

正文应围绕“技术问题、技术手段、技术效果”组织，不应把商业价值、营销表述或管理目标包装为技术贡献。

## 检索与保密规则

网络检索可用时，应使用基于当前日期的动态时间窗口：

- 近期学术或开源动态：当前年份和上一年。
- 扩展学术或趋势检索：当前年份和前两年。
- 专利检索：当前年份和前三年，必要时扩大。
- 公开风险排查：当前年份和前三年。

报告中必须写明：

```text
检索截止日期：YYYY-MM-DD
```

检索词应使用抽象后的技术问题词、技术手段词、技术效果词、同义词、英文词、上位词、下位词和 IPC/CPC 分类号线索。

不得暴露：

- 内部项目名。
- 客户名称。
- 未公开完整技术方案。
- 保密参数。
- 源代码。
- 商业敏感部署细节。

## 如何在 Codex 中使用

用户可以自然语言发起请求，skill 会在匹配专利交底相关任务时自动启用。

示例 1：宽泛方向挖掘

```text
我想生成一篇数据库运维方面的专利交底书。
```

预期流程：

1. 询问公开披露风险。
2. 使用通用技术词检索现有技术。
3. 给出三个可申请方向。
4. 让用户选择或调整方向。

示例 2：发明想法转交底书

```text
帮我写一篇多源数据库故障诊断方法的专利交底书。
```

预期流程：

1. 确认公开披露状态。
2. 访谈技术问题、现有技术、缺点、方案和创新点。
3. 输出《专利申请预判报告》。
4. 用户确认后起草交底书。
5. 按固定模板输出正文。

示例 3：已有材料整理

```text
这是我的技术方案，请帮我整理成专利交底书。
```

预期流程：

1. 询问公开披露情况。
2. 将材料映射到五个交底书章节。
3. 标出缺失内容。
4. 一次最多追问三个关键问题。
5. 做授权前景预判后再正式起草。

示例 4：导出 Word

```text
把这份专利交底书导出成 Word。
```

预期流程：

1. 确认已有完整 Markdown 交底书正文。
2. 使用 `scripts/generate_docx.py`。
3. 输出 `.docx` 文件。

## Word 导出

该 skill 内置 Word 生成脚本：

```bash
python <path-to-skill>/scripts/generate_docx.py <input.md> <output.docx>
```

依赖环境：

- Python 3.x。
- `python-docx`。

输出样式：

- A4 页面。
- 1.5 倍行距。
- 中文字体为 `楷体_GB2312`。
- 西文字体为 `Times New Roman`。
- 发明名称居中。
- 自动生成页码。

Mermaid 图不会由脚本自动渲染。生成的 Word 文件会插入占位提示，要求用户手动插入对应流程图或原理图图片。

## 输出要求

交底书默认输出中文 Markdown。

发明名称应简洁、技术化，通常不超过 25 个汉字。避免使用“最佳”“最优”“超级”等夸大词，也不要使用商品名或品牌名。

技术方案应达到可实施层面：

- 方法类发明应使用 `S1`、`S2`、`S3` 等步骤。
- 软件类发明应描述输入、处理、输出、数据结构、模型、规则和异常处理。
- 系统类发明应描述模块、模块功能、数据流和连接关系。
- 附图应先有文字说明，再提供 Mermaid 代码。

## 限制说明

该 skill：

- 不提供法律意见。
- 不保证专利授权。
- 不替代正式查新或专业检索。
- 可能遗漏付费库、需认证数据库、未公开材料或未被搜索引擎收录的文献。
- 不得编造关键技术特征、参数范围、实验数据或公开事实。

信息不足时，应标注“待确认”，而不是自行补造。

## 用户最佳实践

建议用户尽量提供以下信息：

- 技术领域和应用场景。
- 要解决的技术问题。
- 现有方案及其缺点。
- 具体技术步骤、模块、算法、数据结构或系统架构。
- 输入、输出和判断条件。
- 附图、日志、流程图或实现说明。
- 可量化技术效果，例如延迟降低、准确率提升、资源占用降低或故障率降低。
- 是否已经公开披露，以及公开日期和公开渠道。

在 skill 明确说明用途和必要性之前，不建议发送不必要的商业秘密或内部敏感信息。
